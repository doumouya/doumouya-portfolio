[Case-first default] If this turn will implement a NON-TRIVIAL change — a feature, a logic bugfix, a multi-file edit, anything that will produce a commit — then FIRST open a Case: write a spec doc at docs/specs/<slug>.md (title + a one-paragraph plan + numbered acceptance criteria), or case_create when the Cases MCP is wired. Then log plan → key decisions → outcome and move status (in_progress → in_review → done); before done, reconcile any documented surface in the same commit or declare "Docs: n/a — <reason>". Commit on the working branch with `git commit -o <named files>`; NEVER push without the human (the one hard gate). SKIP all of this for trivial one-line / doc / typo fixes, read-only investigation, and pure Q&A. Reserve the full /feature role-chain (architect → tester → coder → reviewer → ops) for BIG or RISKY features.
